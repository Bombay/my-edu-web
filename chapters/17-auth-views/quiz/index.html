회원가입/로그인 화면 — v-model · POST · API 모듈 · useRouter

1v-model이 하는 일은?

<input v-model="email" />
데이터 → 화면 방향으로만 동기화 (단방향)
화면 → 데이터 방향으로만 동기화 (단방향)
데이터 ↔ 화면 양방향 동기화 — 사용자가 치면 ref가 바뀌고, ref를 바꾸면 input이 바뀐다
input의 값을 서버로 자동 전송한다
정답! v-model은 양방향 바인딩(two-way binding)입니다. input에 뭘 치든 ref.value가 자동으로 바뀌고, 코드에서 ref.value를 바꾸면 input 안의 글자도 자동으로 바뀌어요. 거울처럼요!
오답! v-model은 양방향이에요. {{ }} 는 데이터→화면 단방향이고, @input 같은 이벤트는 화면→데이터 단방향이지만, v-model은 이 둘을 합친 양방향 동기화예요.

2GET 요청과 POST 요청의 가장 큰 차이는?

GET은 느리고 POST는 빠르다
GET은 데이터를 가져오고, POST는 데이터를 보낸다 (body에 담아서)
GET은 브라우저에서만, POST는 서버에서만 사용할 수 있다
GET은 JSON만, POST는 HTML만 처리할 수 있다
정답! GET은 "줘" (서버에서 데이터를 가져옴), POST는 "받아" (서버에 데이터를 보냄)예요. POST에는 method, headers, body를 추가로 지정해야 해요.
오답! GET은 서버에서 데이터를 "가져오는" 요청, POST는 서버에 데이터를 "보내는" 요청이에요. POST는 body에 데이터를 담아 보내고, Content-Type 헤더로 형식을 알려줘야 해요.

3다음 코드에서 @submit.prevent의 .prevent가 하는 일은?

<form @submit.prevent="handleRegister">
브라우저의 기본 폼 제출 동작(페이지 새로고침)을 막는다
폼 제출을 완전히 차단하여 handleRegister가 실행되지 않게 한다
폼 입력값이 비어있으면 제출을 막는다
서버 에러가 발생하면 제출을 취소한다
정답! 브라우저는 원래 form을 제출하면 페이지 전체를 새로고침해요. SPA에서는 이러면 안 되니까 .prevent로 기본 동작을 막고, 대신 우리가 정의한 handleRegister 함수가 JavaScript로 처리하는 거예요.
오답! .prevent는 event.preventDefault()를 자동 호출하는 Vue의 이벤트 수식어예요. 폼의 기본 동작(페이지 새로고침)을 막고, 대신 JavaScript 함수가 요청을 처리하게 해요. SPA에서 필수적인 패턴이에요.

4API 모듈(api/index.ts)을 따로 만드는 가장 큰 이유는?

// 각 컴포넌트에서 이렇게 쓰던 걸:
await fetch('http://localhost:3000/api/boards')
await fetch('http://localhost:3000/api/auth/login', { ... })

// 이렇게 바꿈:
await api('/boards')
await api('/auth/login', { ... })
보안을 위해 URL을 숨기려고
fetch보다 api 함수가 더 빠르게 동작해서
TypeScript가 반드시 모듈로 분리하도록 강제해서
BASE_URL, 헤더, 에러 처리를 한 곳에서 관리하여 중복을 없애고 변경 시 한 곳만 수정하면 되니까
정답! 도메인이 바뀌면 BASE_URL 한 줄만 수정, 헤더 추가도 한 곳에서, 에러 처리 패턴도 한 곳에서. 10개 컴포넌트에서 같은 코드를 반복하지 않아도 돼요. 이런 "공통 로직 한 곳에 모으기"는 개발의 기본 원칙이에요.
오답! 핵심은 "중복 제거 + 변경 용이성"이에요. URL, 헤더, 에러 처리를 한 곳에서 관리하면 수정할 때 한 파일만 고치면 되고, 각 컴포넌트 코드도 간결해져요.

5다음 코드에서 const data = await api(...)로 값을 받는 이유는?

// 회원가입 — 값을 안 받음
await api('/auth/register', { ... })

// 로그인 — 값을 받음
const data = await api('/auth/login', { ... })
console.log(data.token)
회원가입 API는 return이 없고, 로그인 API만 return이 있어서
const를 붙이지 않으면 에러가 나서
로그인 응답에는 토큰이 포함되어 있어서, 나중에 사용하려면 변수에 받아둬야 하니까
await는 반드시 const로 받아야 동작한다
정답! 함수가 return하는 값은 받아도 되고 안 받아도 돼요. 회원가입은 "성공 여부"만 알면 되니까 안 받았고, 로그인은 토큰을 나중에 써야 하니까 변수에 받은 거예요. 필요에 따라 선택하면 됩니다.
오답! await api(...)는 값을 받든 안 받든 동작해요. 차이는 "그 응답을 나중에 쓸 일이 있느냐"예요. 로그인은 토큰(data.token)을 저장해야 하니까 변수로 받은 거고, 회원가입은 응답 데이터가 딱히 필요 없어서 안 받은 거예요.

6router.push('/login')은 어떤 상황에서 사용하나요?

사용자가 링크를 클릭할 때 (템플릿에서 페이지 이동)
JavaScript 코드 안에서 프로그래밍적으로 페이지를 이동할 때 (예: 회원가입 성공 후 자동 이동)
브라우저 뒤로가기 버튼을 누를 때
서버에서 리다이렉트 응답을 보낼 때
정답! RouterLink는 템플릿(HTML)에서 클릭으로 이동할 때, router.push()는 JavaScript 코드 안에서 조건에 따라 이동할 때 써요. "회원가입 성공하면 로그인 페이지로" 같은 로직은 코드에서 처리해야 하니까 router.push()가 필요해요.
오답! 링크 클릭은 <RouterLink>의 영역이에요. router.push()는 코드 안에서 "어떤 조건이 충족되면 페이지를 이동시켜라" 할 때 써요. 대표 예시: 회원가입/로그인 성공 후 자동 이동.
