Chapter 13 Quiz

에러 처리와 검증 - 통합 에러 미들웨어

1Express가 에러 처리 미들웨어를 구분하는 방법은?

// 일반 미들웨어
function middleware(req, res, next) { ... }

// 에러 미들웨어
function errorMiddleware(err, req, res, next) { ... }
함수 이름에 "error"가 포함되어 있으면
app.useError()로 등록하면
매개변수가 4개(err, req, res, next)이면
미들웨어 안에서 try-catch를 쓰면
정답! Express는 매개변수가 4개인 미들웨어를 에러 처리용으로 인식합니다. 첫 번째 매개변수 err에 에러 객체가 전달됩니다.
오답! 함수 이름은 상관없습니다. Express는 오직 매개변수 개수(4개: err, req, res, next)로 에러 미들웨어를 구분합니다.

2에러 미들웨어를 등록하는 올바른 위치는?

const app = express();
app.use(express.json());
// A: 여기?
app.use('/api/posts', postsRouter);
app.use('/api/auth', authRouter);
// B: 여기?
app.listen(3000);
A — 라우터보다 위에
B — 라우터보다 아래에
A든 B든 상관없다
app.listen() 안에 넣어야 한다
정답! 에러 미들웨어는 라우터 아래에 위치해야 합니다. 라우터에서 에러가 발생하면 아래로 흘러내려와서 에러 미들웨어에 도착하는 구조입니다.
오답! 에러 미들웨어가 라우터보다 위에 있으면, 라우터에서 발생한 에러가 도달하지 못합니다. 반드시 라우터 아래에 등록해야 해요.

3다음 코드에서 next(err)의 역할은?

router.get('/:id', (req, res, next) => {
    try {
        const post = db.prepare('SELECT * FROM posts WHERE id = ?').get(req.params.id);
        if (!post) {
            throw new NotFoundError('글을 찾을 수 없습니다');
        }
        res.json(post);
    } catch (err) {
        next(err);  // ???
    }
});
다음 라우터로 요청을 넘긴다
에러를 무시하고 계속 진행한다
에러 메시지를 콘솔에 출력한다
잡은 에러를 에러 미들웨어로 전달한다
정답! next()에 에러 객체를 넣으면 Express가 에러 미들웨어(매개변수 4개)를 찾아서 전달합니다. 화재 경보를 울려서 중앙 관제실로 보내는 것과 같아요.
오답! next()만 호출하면 다음 미들웨어로 넘기지만, next(err)처럼 에러를 넣으면 에러 미들웨어로 전달됩니다. 일반 미들웨어는 건너뜁니다.

4커스텀 에러 클래스를 사용하는 이유로 가장 적절한 것은?

// 이전
res.status(404).json({ error: '글을 찾을 수 없습니다' });

// 이후
throw new NotFoundError('글을 찾을 수 없습니다');
상태 코드를 외울 필요 없이 이름만으로 의미가 통하고, 에러 형식이 통일된다
코드 실행 속도가 더 빨라진다
try-catch 없이도 에러를 처리할 수 있다
프론트엔드에서 에러를 자동으로 처리해준다
정답! NotFoundError만 봐도 "404구나" 알 수 있고, 에러 미들웨어가 통일된 형식으로 응답합니다. 코드 가독성과 일관성이 모두 좋아지죠.
오답! 성능이나 자동 처리와는 관계없습니다. 핵심은 코드 가독성(이름만으로 의미 전달)과 응답 형식의 통일입니다.

5예상치 못한 에러(버그, DB 장애 등)가 발생했을 때, 서버가 사용자에게 보내야 하는 응답은?

에러의 스택 트레이스를 그대로 보여준다
DB 연결 정보와 함께 에러 원인을 알려준다
500 상태 코드와 "서버 내부 오류" 같은 일반적인 메시지만 보낸다
아무 응답도 보내지 않는다
정답! 내부 에러 정보(스택 트레이스, DB 정보 등)를 사용자에게 보여주면 보안 위험이 됩니다. 사용자에겐 일반적인 메시지만 보내고, 상세 에러는 서버 로그에만 기록합니다.
오답! 에러의 상세 정보(스택 트레이스, DB 정보)를 사용자에게 보여주면 공격자에게 힌트를 주는 셈입니다. 500 상태 코드와 일반적인 메시지만 보내고, 상세 로그는 서버에만 남깁니다.
