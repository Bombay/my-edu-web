Chapter 18 Quiz

인증 상태 관리 — Pinia · localStorage · 라우트 가드

1Pinia 스토어가 필요한 이유는?

ref가 너무 느려서 더 빠른 상태 관리가 필요하다
여러 컴포넌트가 같은 데이터(예: 로그인 토큰)를 공유해야 하는데, 각 컴포넌트의 ref는 자기 안에서만 존재하기 때문
Vue가 Pinia 없이는 ref를 사용할 수 없어서
서버와 통신하려면 반드시 Pinia를 거쳐야 해서
정답! ref는 컴포넌트 안에서만 사는 "책상 서랍"이에요. LoginView에서 받은 토큰을 App.vue 네비나 BoardsView에서도 쓰려면 "공용 사물함"(Pinia 스토어)이 필요해요.
오답! 핵심은 "데이터 공유"예요. 각 컴포넌트의 ref는 자기 안에서만 존재하므로, 여러 컴포넌트가 같은 데이터를 봐야 할 때 Pinia 같은 전역 상태 관리가 필요합니다.

2ref와 localStorage의 차이는?

ref는 문자열만, localStorage는 객체도 저장할 수 있다
ref는 서버에, localStorage는 브라우저에 저장된다
ref는 메모리에 저장되어 새로고침하면 사라지고, localStorage는 디스크에 저장되어 새로고침해도 유지된다
둘 다 같은 곳에 저장되지만 문법만 다르다
정답! ref는 JavaScript 메모리(화이트보드) — 새로고침하면 지워져요. localStorage는 브라우저 디스크(서랍 메모지) — 새로고침, 브라우저 닫기에도 살아남아요. 그래서 둘 다에 저장하면 반응성(ref) + 영속성(localStorage) 모두 챙길 수 있어요.
오답! ref는 JavaScript 메모리에 저장돼서 새로고침하면 사라져요. localStorage는 브라우저의 영구 저장소에 저장돼서 새로고침이나 브라우저를 닫아도 유지돼요. 참고로 localStorage는 문자열만 저장 가능하고, ref는 어떤 타입이든 가능해요.

3다음 코드에서 loadFromStorage()를 App.vue에서 호출하는 이유는?

// App.vue
const authStore = useAuthStore()
authStore.loadFromStorage()  // ← 이걸 왜 여기서?
Pinia가 자동으로 localStorage를 읽지 못해서 에러가 나기 때문
LoginView에서는 호출할 수 없기 때문
서버에서 토큰을 다시 발급받기 위해
새로고침하면 ref(메모리)가 비워지므로, 앱 시작 시 localStorage에서 토큰을 꺼내 ref에 복원해야 로그인 상태를 유지할 수 있으니까
정답! 새로고침하면 JavaScript 메모리(ref)는 싹 비워져요. 하지만 localStorage에는 이전에 저장한 토큰이 남아있어요. 앱이 시작될 때 loadFromStorage()로 그걸 다시 ref에 복원하면 로그인 상태가 유지돼요.
오답! 새로고침하면 ref(메모리)는 초기화돼요. localStorage에 남아있는 토큰을 앱 시작 시 ref에 복원해줘야 로그인 상태가 유지됩니다. App.vue는 모든 페이지에서 항상 실행되므로 여기서 호출하는 게 적절해요.

4api 모듈에서 Authorization: Bearer ${token} 헤더를 보내는 이유는?

서버의 authMiddleware가 이 헤더를 확인하여 "로그인한 사용자"인지 검증하기 때문
CORS 에러를 방지하기 위해
localStorage에 접근하기 위한 권한을 얻기 위해
브라우저가 이 헤더 없이는 fetch를 거부하기 때문
정답! 게시판 생성, 글쓰기 같은 인증이 필요한 API는 서버의 authMiddleware가 Authorization 헤더에서 JWT 토큰을 꺼내 검증해요. "이 요청을 보낸 사람이 누구인지" 확인하는 신분증 같은 거예요.
오답! Authorization 헤더는 서버에게 "나 로그인한 사람이에요"를 증명하는 용도예요. 서버의 authMiddleware(Chapter 11에서 만든 것)가 이 헤더에서 JWT 토큰을 꺼내 검증합니다.

5라우트 가드 router.beforeEach에 대한 설명 중 맞는 것은?

router.beforeEach((to) => {
  const token = localStorage.getItem('token')
  if (to.meta.requiresAuth && !token) {
    return '/login'
  }
})
meta.requiresAuth가 없는 페이지에도 토큰을 검사하여 모든 페이지를 차단한다
모든 페이지 이동 전에 실행되지만, meta.requiresAuth가 true이고 토큰이 없을 때만 /login으로 보낸다
서버에 토큰 유효성을 확인한 뒤 결과에 따라 차단한다
한 번만 실행되고 이후 페이지 이동에는 영향을 주지 않는다
정답! beforeEach는 "모든" 페이지 이동 전에 실행되지만, if 조건 덕분에 requiresAuth가 true인 페이지에서만 토큰을 확인해요. 홈, 로그인, 회원가입 페이지는 meta가 없으니까 자유롭게 통과합니다.
오답! beforeEach는 매 페이지 이동마다 실행돼요. 하지만 if 조건문 때문에 실제로 차단되는 건 meta.requiresAuth가 true이고 토큰이 없을 때뿐이에요. 그 외 페이지는 자유롭게 접근 가능합니다.

6다음 두 줄의 관계는?

import { useAuthStore } from '../stores/auth'
const authStore = useAuthStore()
두 줄은 같은 일을 하는 중복 코드다
useAuthStore는 스토어의 데이터이고, authStore는 그 복사본이다
useAuthStore는 스토어를 여는 "열쇠"(함수)이고, 호출해야 실제 스토어 객체(authStore)를 얻는다
import는 서버에서 데이터를 가져오고, const는 로컬에 저장한다
정답! import로 가져오는 useAuthStore는 함수(열쇠)예요. 이 함수를 호출(괄호!)해야 실제 스토어 객체가 반환돼요. 변수 이름(authStore)은 자유롭게 정할 수 있어요.
오답! useAuthStore는 함수예요. import로 가져오기만 해서는 스토어를 쓸 수 없고, 괄호를 붙여 호출해야(useAuthStore()) 실제 스토어 객체를 얻어요. "열쇠를 가져온 것"과 "열쇠로 문을 연 것"의 차이예요.
