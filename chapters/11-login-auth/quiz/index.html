1JWT의 3가지 구성 요소는?

eyJhbGciOi...  .  eyJ1c2VySW...  .  TyNOeg-P4T...
이름 . 비밀번호 . 만료시간
암호화키 . 데이터 . 체크섬
헤더 . 페이로드 . 서명
요청 . 응답 . 인증코드
정답! 헤더(알고리즘 정보), 페이로드(담고 싶은 데이터), 서명(위조 방지)으로 구성됩니다.
오답! JWT는 헤더(Header), 페이로드(Payload), 서명(Signature) 3부분으로 이루어져 있습니다. 점(.)으로 구분되죠.

2로그인 실패 시 "이메일이 존재하지 않습니다" 대신 "이메일 또는 비밀번호가 틀렸습니다"라고 응답하는 이유는?

코드를 간단하게 쓰기 위해서
공격자에게 어떤 이메일이 가입되어 있는지 힌트를 주지 않기 위해서
HTTP 표준에서 그렇게 하라고 정해놨기 때문에
사용자가 혼란스럽지 않도록 하기 위해서
정답! "이메일이 없습니다"라고 하면 공격자가 가입된 이메일 목록을 알아낼 수 있습니다. 보안을 위해 의도적으로 모호한 메시지를 사용합니다.
오답! 이건 보안 때문입니다. "이메일이 없습니다"라고 구체적으로 알려주면, 공격자가 이메일을 하나씩 넣어보며 가입된 계정 목록을 수집할 수 있어요.

3미들웨어에서 next()를 호출하는 것은 어떤 의미인가?

export function authMiddleware(req, res, next) {
    // 토큰 검증 성공 시
    req.user = decoded;
    next();  // ???
}
응답을 클라이언트에게 보낸다
에러가 발생했음을 알린다
미들웨어를 처음부터 다시 실행한다
검증을 통과시키고 다음 단계(실제 API 핸들러)로 넘긴다
정답! next()는 "검사 끝, 다음으로 넘겨!"라는 뜻입니다. 검표원이 팔찌를 확인하고 통과시키는 것과 같습니다.
오답! next()는 현재 미들웨어의 역할이 끝났으니 다음 단계(라우터 핸들러)로 요청을 넘기라는 의미입니다. 놀이기구 앞 검표원이 통과시키는 것과 같아요.

4다음 코드에서 authMiddleware의 위치가 올바른 것은?

router.post('/', authMiddleware, (req, res) => { ... })
router.post('/', (req, res) => { authMiddleware(); ... })
authMiddleware(router.post('/', (req, res) => { ... }))
router.post('/', (req, res) => { ... }, authMiddleware)
정답! 미들웨어는 라우트 경로와 핸들러 사이에 넣습니다. Express가 순서대로 실행하기 때문에 미들웨어가 먼저 실행되고, next()를 호출하면 핸들러로 넘어갑니다.
오답! Express에서 미들웨어는 경로('/')와 핸들러 함수 사이에 넣습니다. router.post('/', authMiddleware, handler) 형태로요. Express가 왼쪽에서 오른쪽으로 순서대로 실행합니다.

5글 작성 시 req.user.email을 author로 사용하는 이유는?

// 이전: body에서 author를 받음
const { title, content, author } = req.body;

// 이후: 토큰에서 email을 꺼냄
const result = db.prepare(
  'INSERT INTO posts (title, content, author) VALUES (?, ?, ?)'
).run(title, content, req.user.email);
코드가 더 짧아지니까
body에서 email을 보내는 것이 불가능하니까
토큰은 서버가 검증한 정보라 사칭이 불가능하니까
body에 author 필드를 추가하면 에러가 발생하니까
정답! body의 데이터는 누구든 마음대로 보낼 수 있지만, 토큰은 서버가 발급하고 서명한 것이라 위조할 수 없습니다. 그래서 토큰에서 꺼낸 정보를 신뢰합니다.
오답! 핵심은 신뢰의 문제입니다. body는 누구든 조작할 수 있지만(author: "관리자"처럼), 토큰은 서버의 비밀 키로 서명되어 있어서 위조가 불가능합니다.
