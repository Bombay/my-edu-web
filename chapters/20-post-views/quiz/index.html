1/boards/3 페이지에서 3번 게시판의 글만 가져오려면?

const route = useRoute()
const boardId = route.params.id  // "3"
// ↓ 빈칸에 들어갈 코드는?
const posts = await api(______)
'/posts' — 어차피 모든 글이 필요하니까
'/posts/${boardId}' — 따옴표 안에 변수를 넣음
`/posts?boardId=${boardId}` — 백틱으로 템플릿 리터럴, 쿼리 파라미터로 게시판 번호 전달
'/posts'.boardId — 점 표기법으로 필터
정답! 백틱(``)과 ${}로 변수를 URL에 끼워넣고, ?boardId=3처럼 쿼리 파라미터로 보내면 백엔드가 그 게시판 글만 골라줘요. 일반 따옴표('')에서는 ${}가 동작하지 않아요.
오답! 일반 따옴표('') 안에서는 ${}가 동작하지 않아 그대로 문자열로 들어가요. 변수를 URL에 끼워넣으려면 반드시 백틱(``)을 써야 합니다. 그리고 "3번 게시판의 글만"을 요청하려면 쿼리 파라미터(?boardId=3)가 필요해요.

2Express에서 다음 두 값을 꺼내는 방법으로 올바른 짝은?

// 요청 1: GET /api/posts/5
// 요청 2: GET /api/posts?boardId=3
둘 다 req.params에서 꺼낸다
요청 1은 req.params.id, 요청 2는 req.query.boardId
요청 1은 req.query.id, 요청 2는 req.params.boardId
둘 다 req.body에서 꺼낸다
정답! URL 경로의 :id 같은 자리는 req.params로, ?key=value 형태의 쿼리 문자열은 req.query로 꺼내요. 헷갈리는 두 통로를 정확히 구분하는 게 중요합니다.
오답! 두 통로를 헷갈리면 안 됩니다. URL 경로의 /:id는 자원을 식별하는 부분이라 req.params로, ?로 시작하는 쿼리 문자열은 선택적 옵션/필터라 req.query로 꺼냅니다. 그래서 라우트 정의도 다르죠 — router.get('/:id', ...) vs router.get('/', ...).

3본문(content)을 입력받는 가장 적절한 방법은?

const newContent = ref('')
// 본문은 여러 줄 입력이 필요함
<input v-model="newContent" type="long" />
<textarea v-model="newContent"></textarea>
<div v-model="newContent" contenteditable></div>
input을 여러 개 만들어서 한 줄씩 받는다
정답! 여러 줄 입력은 textarea가 표준이고, v-model은 input과 똑같이 그냥 붙이면 됩니다. Vue가 내부적으로 적절한 속성에 알아서 연결해줘요. type="long" 같은 속성은 존재하지 않아요.
오답! input의 type 속성에 "long"은 없어요. 여러 줄 입력의 표준 태그는 textarea이고, v-model은 input/textarea/select 모두 동일하게 사용 가능합니다. div + contenteditable은 v-model이 기본 지원하지 않아 별도 처리가 필요해요.

4새 글 작성 후 목록 맨 위에 즉시 보이게 하려면? (배열 앞에 추가)

const newPost = await api('/posts', { method: 'POST', body: ... })
// ↓ 이 자리에 들어갈 코드
posts.value.push(newPost)
posts.value = newPost
posts.value.unshift(newPost)
posts.value.add(newPost)
정답! push는 배열의 맨 뒤에 추가, unshift는 맨 앞에 추가합니다. 최신 글이 위로 와야 자연스러우면 unshift를 씁니다. 둘 다 Vue 반응성을 자동으로 트리거해서 화면이 즉시 갱신돼요.
오답! push는 맨 뒤에 추가라 새 글이 목록 끝에 묻혀요. posts.value = newPost는 객체로 덮어써서 배열이 깨지고, JavaScript 배열에는 add 메서드가 없습니다. 정답은 unshift — 배열 앞에 추가.

5이번에 만난 버그: 백엔드가 { error: "글을 찾을 수 없습니다" }를 보내는데, 화면에는 "서버 오류: 404"가 떴습니다. 원인은?

// 백엔드
res.status(404).json({ error: '글을 찾을 수 없습니다' })

// 프론트 api 모듈
throw new Error(errorData?.message || `서버 오류: ${response.status}`)
백엔드가 status 404를 잘못 설정했다
404는 원래 메시지를 못 받는다
백엔드는 error 키로 보내는데, 프론트는 message 키를 찾고 있어 undefined가 되어 폴백 메시지가 표시됨 — 양쪽의 JSON 키 약속이 어긋난 것
JWT 토큰이 만료되어서
정답! 컴파일 에러도 안 나고 조용히 잘못 동작하는 까다로운 버그예요. 프론트와 백엔드는 JSON의 키 이름까지 정확히 맞춰야 통합니다. 실무에서도 가장 자주 만나는 부류의 버그라 디버깅 도구(Network 탭)에 익숙해지는 게 중요해요.
오답! status는 404가 맞고, JWT 만료와도 무관해요. 핵심은 "프론트와 백엔드가 같은 JSON 모양에 합의해야 한다"는 점입니다. 백엔드가 보낸 키(error)와 프론트가 찾는 키(message)가 달라서, errorData.message가 undefined가 됐고 폴백인 "서버 오류: 404"가 표시된 거예요.
