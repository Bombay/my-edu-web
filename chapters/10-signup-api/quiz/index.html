회원가입 API - 비밀번호 해싱과 입력 검증

1비밀번호를 DB에 저장할 때, 올바른 방법은?

원본 비밀번호를 그대로 TEXT 컬럼에 저장한다
bcrypt.hashSync()로 해싱한 결과를 저장한다
비밀번호를 Base64로 인코딩해서 저장한다
비밀번호를 거꾸로 뒤집어서 저장한다
정답! 해싱은 되돌릴 수 없는 변환이라 DB가 유출되어도 원본 비밀번호를 알 수 없습니다. Base64나 뒤집기는 쉽게 원본을 복원할 수 있어요.
오답! Base64 인코딩이나 뒤집기는 쉽게 원본으로 되돌릴 수 있고, 원본 저장은 유출 시 큰 피해로 이어집니다. bcrypt 해싱은 되돌릴 수 없어서 안전합니다.

2같은 비밀번호 "abc123"을 bcrypt로 두 번 해싱하면?

const hash1 = bcrypt.hashSync("abc123", 10);
const hash2 = bcrypt.hashSync("abc123", 10);
console.log(hash1 === hash2); // ???
true — 같은 입력이니 항상 같은 결과
에러 발생 — 같은 비밀번호를 두 번 해싱할 수 없다
false — 매번 랜덤 salt가 섞여서 다른 결과가 나온다
때로는 true, 때로는 false — 랜덤이라 알 수 없다
정답! 아까 직접 실험해서 확인했죠? bcrypt는 매번 랜덤 salt를 생성하기 때문에 같은 비밀번호라도 다른 해시값이 나옵니다. 그래서 비교할 때는 compareSync()를 사용합니다.
오답! 아까 실험 결과를 떠올려보세요. hash1 === hash2는 false였습니다. bcrypt는 매번 랜덤 salt를 섞기 때문입니다.

3다음 회원가입 API 응답에서 잘못된 점은?

// POST /api/auth/register 응답
{
  "id": 1,
  "email": "user@test.com",
  "nickname": "유저",
  "password_hash": "$2b$10$XApl...",
  "created_at": "2026-04-02 12:00:00"
}
password_hash가 응답에 포함되면 안 된다
id가 숫자가 아니라 UUID여야 한다
created_at 형식이 잘못되었다
문제 없다. 어차피 해싱되었으니 안전하다
정답! 해싱되었더라도 password_hash를 응답에 포함하면 안 됩니다. 해시값으로도 공격이 가능하고, 불필요한 민감 정보는 최소한으로 노출하는 것이 보안의 원칙입니다.
오답! 해싱되었다고 안전한 것이 아닙니다. password_hash도 민감 정보이므로 응답에 절대 포함하면 안 됩니다. SELECT할 때 필요한 컬럼만 선택하세요.

4이미 가입된 이메일로 다시 회원가입을 시도하면, 적절한 HTTP 상태 코드는?

400 Bad Request
401 Unauthorized
404 Not Found
409 Conflict
정답! 409 Conflict는 "요청은 이해했지만, 현재 서버 상태와 충돌한다"는 의미입니다. 이미 존재하는 리소스와 충돌하는 상황에 딱 맞습니다.
오답! 400은 요청 형식 오류, 401은 인증 필요, 404는 리소스 없음입니다. 이미 존재하는 데이터와 충돌하는 상황에는 409 Conflict가 적절합니다.

5다음 코드에서 입력 검증 순서로 가장 적절한 것은?

router.post('/register', (req, res) => {
    const { email, nickname, password } = req.body;
    // A: 이메일 중복 확인 (DB 조회)
    // B: 필수 항목 체크 (빈 값)
    // C: 비밀번호 길이 확인
    // D: 이메일 형식 확인
});
A → B → C → D
B → D → C → A
D → C → A → B
순서는 상관없다
정답! 빠르고 가벼운 검증부터 먼저 합니다. 빈 값(B) → 형식(D) → 길이(C) → DB 조회(A). 빈 값인데 굳이 DB를 조회할 필요가 없겠죠? DB 조회는 비용이 크니 마지막에!
오답! 검증은 가벼운 것부터 무거운 것 순으로 합니다. 빈 값 체크(B)가 가장 가볍고, DB 중복 조회(A)가 가장 무겁습니다. 빈 값인데 DB를 조회하면 낭비겠죠?
