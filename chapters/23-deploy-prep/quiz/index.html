1다음 중 .env 파일에 넣어야 할 값으로 가장 적절한 묶음은?

JWT 시크릿, 사용자 닉네임, UI 색상
UI 텍스트, CSS 변수, 버튼 라벨
JWT 시크릿, DB 접속 비밀번호, 서버 포트, API 베이스 URL — 비밀이거나 환경마다 다른 값
컴포넌트 이름, 라우트 경로, 함수명
정답! .env에는 두 종류가 들어가요 — ① 비밀(시크릿, DB 비밀번호 등 코드/Git에 노출되면 안 되는 것), ② 환경마다 다른 값(주소, 포트, 모드). UI 텍스트나 컴포넌트명은 코드의 일부로 .env와는 무관해요.
오답! .env는 "비밀" 또는 "환경에 따라 다른 값"을 담아요. UI 텍스트, 색, 컴포넌트명 같은 건 코드의 일부일 뿐이라 .env와 상관없어요. 사용자 닉네임은 DB에 저장되는 데이터고요.

2Vite 프론트엔드에서 환경변수 이름이 반드시 VITE_로 시작해야 하는 이유는?

Vite 라이브러리가 다른 이름은 알파벳 순으로 정렬할 수 없어서
프론트엔드 코드는 사용자 브라우저로 노출되므로, "이 변수는 노출돼도 안전하다"고 명시한 것만 통과시키는 보안 장치
JavaScript 변수명은 모두 VITE_로 시작해야 하는 표준이라서
Vite 5.0부터 생긴 임의 규칙으로 의미 없음
정답! 프론트엔드 빌드 결과는 사용자 브라우저로 그대로 다운로드돼요. 그래서 .env의 모든 값을 자동 노출하면 비밀번호/시크릿이 박힐 위험이 큼. Vite는 개발자가 명시적으로 "이건 공개 OK"라고 표시한 VITE_ 변수만 노출시켜 사고를 막아요.
오답! 핵심은 보안이에요. 프론트엔드 코드는 사용자 브라우저에 그대로 다운로드되므로, .env의 비밀이 통째로 노출되면 큰일납니다. Vite는 "VITE_로 시작하는 변수만 빌드에 포함"시키는 강제 규칙으로 사고를 막아요. JavaScript 표준이나 Vite 임의 규칙이 아니라 보안 장치입니다.

3.env.example을 따로 만들고 Git에 커밋하는 이유는?

.env가 깨졌을 때 자동 복구되도록
진짜 .env는 Git에 안 올라가니, 팀원/미래의 나에게 "어떤 변수가 필요한지" 알려주는 템플릿 역할
Vite가 production 빌드 시 자동으로 사용
필수 파일이라 없으면 npm run build가 실패
정답! .env는 비밀이 들어있어 Git에 안 올라가요(.gitignore). 그러면 새 팀원이 프로젝트를 받았을 때 "어떤 변수를 채워야 하지?"를 알 길이 없어요. .env.example은 키 이름만 있는 빈 템플릿으로 그 역할을 합니다. 관례적으로 .env와 짝을 이뤄 사용해요.
오답! .env.example은 빌드와 무관하고 자동 복구 기능도 없어요. 핵심 목적은 "필요한 환경변수 키 이름을 팀에 공유하는 템플릿". .env는 Git에 안 올라가니 그것만 봐선 어떤 변수가 필요한지 알 수 없어서, 빈 값의 .env.example을 같이 커밋해두는 관례예요.

4빌드(npm run build)가 만들어내는 것은?

새 npm 패키지를 자동 다운로드
서버를 자동 시작
dist/ 폴더에 실행 환경(Node.js / 브라우저)이 직접 읽을 수 있는 형태(JS/HTML/CSS)로 변환된 산출물 — 배포 시 이걸 올림
코드를 클라우드에 자동 업로드
정답! 빌드는 우리가 쓴 TS/Vue를 실행 환경이 직접 읽을 수 있는 JS/HTML/CSS로 사전 변환합니다. 결과는 dist/ 폴더. 배포 서비스(Vercel 등)는 npm install 후 npm run build를 돌리고, 그 dist/를 호스팅해요. 자동 업로드/서버 시작은 별개 도구의 역할.
오답! 빌드는 코드 변환 작업이에요 — TS는 JS로, Vue는 JS+HTML+CSS로. 결과는 dist/ 폴더에 떨어지고, 배포 시 그 dist/를 서버에 올립니다. 패키지 다운로드는 npm install, 서버 시작은 npm start, 클라우드 업로드는 vercel deploy 등 별도 명령이에요.

5SQLite의 다음 컬럼을 PostgreSQL(Supabase)로 옮긴다면 가장 적절한 변환은?

id INTEGER PRIMARY KEY AUTOINCREMENT,
created_at DATETIME DEFAULT CURRENT_TIMESTAMP
그대로 둔다 — SQL은 표준이라 모든 DB에서 동일 동작
id SERIAL PRIMARY KEY, created_at TIMESTAMPTZ DEFAULT NOW()
id NUMBER, created_at STRING
컬럼을 다 지우고 PostgreSQL이 자동 생성하게 둔다
정답! PostgreSQL은 자동증가 정수 전용 타입 SERIAL을 제공해요(INTEGER + AUTOINCREMENT 합친 격). 시간은 시간대 정보가 포함된 TIMESTAMPTZ가 권장이고, 현재 시각 함수는 NOW()입니다. SQL은 표준이 있지만 DB마다 방언이 있어 작은 변환이 필요해요.
오답! SQL은 ANSI 표준이 있긴 하지만 각 DB마다 자신만의 타입과 함수(방언)가 있어요. 그래서 DB를 바꾸면 작은 변환이 필요합니다. PostgreSQL은 SERIAL(자동증가 정수), TIMESTAMPTZ(시간대 포함 타임스탬프), NOW()(현재 시각) 같은 자기만의 표준을 씁니다. NUMBER/STRING은 PG에 없는 타입이에요.
