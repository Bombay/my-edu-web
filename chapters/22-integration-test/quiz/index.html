Chapter 22 Quiz

통합 테스트 — JWT · 미들웨어 · 에러 미들웨어 · PUT/PATCH

1회원가입 API와 로그인 API를 각각 단위 테스트로 점검했을 때 모두 통과했습니다. 그런데 "회원가입 직후 같은 계정으로 로그인" 시나리오에서 실패가 발견됐습니다. 이런 종류의 버그를 가장 잘 잡아내는 테스트는?

단위 테스트를 더 꼼꼼히 만든다 — 각 함수의 모든 입력 케이스를 검증하면 모든 버그가 잡힌다
정적 분석(타입 체크)을 강화한다 — TypeScript가 모든 런타임 버그를 막아준다
통합 테스트로 사용자 시나리오 전체(회원가입 → 로그인 → 글쓰기)를 끝~끝으로 점검한다 — 부분 사이의 약속 어긋남을 잡는다
QA에 맡기고 개발자는 단위 테스트만 책임진다
정답! 각 단위는 멀쩡한데 합쳐지면 깨지는 버그(예: 회원가입은 평문 저장, 로그인은 해시 비교)는 단위 테스트로 절대 안 잡혀요. 통합 테스트가 잡는 영역이 바로 "부분 사이의 합의가 어긋난 부분"이에요.
오답! 단위 테스트만 늘려도 "각 부분은 잘 동작하는데 합치니 안 됨" 같은 버그는 못 잡아요. 타입 체크는 형식 오류만 잡지 약속(회원가입과 로그인이 같은 비밀번호 처리 방식을 쓰는지)은 검증 못합니다. 통합 테스트만이 "흐름 위에서만 드러나는 버그"를 잡을 수 있어요.

2JWT 토큰의 가운데 부분(페이로드)을 atob()로 디코드하면 {userId, email, exp}가 그대로 보입니다. 그렇다면 누가 페이로드의 userId를 다른 사람 ID로 바꿔서 서버에 보내면 그 사람으로 위장할 수 있을까요?

위장할 수 있다 — 페이로드가 디코드 가능한 이상 위변조도 막을 방법이 없다
위장할 수 없다 — 토큰의 마지막 부분(서명)이 서버만 아는 비밀키로 만들어져서, 페이로드를 바꾸면 서명이 안 맞아 거부된다
위장할 수 있다 — JWT는 원래 보안용이 아니라 그냥 데이터 전달용이다
위장할 수 없다 — 페이로드가 사실 암호화되어 있어서 디코드한 게 진짜 값이 아니다
정답! JWT 페이로드는 암호화가 아니라 인코딩(Base64)이라 누구나 볼 수 있어요. 하지만 마지막 "서명"은 서버의 비밀키로 만들어진 도장이라, 페이로드를 살짝만 바꿔도 서명이 안 맞아 즉시 거부돼요. 그래서 비밀번호 같은 민감 정보는 페이로드에 절대 안 넣지만, 위변조는 서명이 막아줍니다.
오답! 페이로드는 누구나 디코드해서 볼 수 있는 게 맞지만, 그렇다고 위변조가 통하진 않아요. 마지막 "서명"이 서버만 아는 비밀키로 만들어진 도장이라, 페이로드 한 글자만 바꿔도 서명이 어긋나서 거부됩니다. 그리고 페이로드는 암호화가 아니라 인코딩이에요 — 디코드한 값이 진짜 값입니다.

3다음 인증 미들웨어가 토큰이 없는 요청을 받았을 때, 그 뒤의 라우트 핸들러는 어떻게 될까요?

function authMiddleware(req, res, next) {
  const token = req.headers.authorization;
  if (!token) {
    return res.status(401).json({ error: '로그인이 필요합니다' });
  }
  // ... 토큰 검증 후
  next();
}

router.post('/posts', authMiddleware, (req, res) => {
  // 글 저장 코드
});
라우트 핸들러는 실행되지만, 응답이 두 번 나가서 에러가 발생한다
라우트 핸들러는 단 한 줄도 실행되지 않는다 — 미들웨어가 next()를 부르지 않고 401 응답을 보냈기 때문
Express가 자동으로 라우트 핸들러를 호출하므로 글이 저장된다 (보안 구멍!)
라우트 핸들러는 실행되지만 req.user가 없어서 내부에서 에러가 난다
정답! 미들웨어 흐름의 핵심: 다음 단계로 넘기려면 반드시 next()를 명시적으로 호출해야 해요. 미들웨어가 res.status(401).json(...)으로 응답을 끝내고 next()를 안 부르면, 그 뒤의 라우트 핸들러는 실행조차 안 됩니다. 이게 인증 보호의 핵심 동작이에요.
오답! Express는 자동으로 다음 단계로 넘어가지 않아요 — 반드시 next()를 호출해야 합니다. 미들웨어가 next() 없이 응답만 보내면 거기서 흐름이 끝나요. 그래서 라우트 핸들러는 단 한 줄도 실행되지 않고, 응답도 두 번 나가지 않아요. 이게 "미들웨어가 라우트 핸들러를 보호한다"는 말의 진짜 의미입니다.

4아래 두 함수가 모두 app.use(...)로 등록됐을 때, Express는 어느 쪽을 "에러 미들웨어"로 인식할까요? 그 구분 기준은?

function A(req, res, next) {
  console.log('logging');
  next();
}

function B(err, req, res, next) {
  res.status(500).json({ error: err.message });
}
A — 함수 이름이 "Middleware"로 끝나야 에러 미들웨어로 인식된다
A — console.log가 들어 있어서 디버깅용으로 인식된다
B — Express는 매개변수가 4개(err가 첫 번째)인 함수를 자동으로 에러 미들웨어로 인식한다
둘 다 — 등록 순서대로 모든 미들웨어가 에러 처리에 참여한다
정답! Express의 유일한 식별 기준은 매개변수 개수예요. 4개(err, req, res, next)면 자동으로 에러 미들웨어로 인식하고, 라우트 핸들러나 다른 미들웨어에서 next(err)를 호출하면 거기로 점프합니다. 이름이나 위치가 아니라 시그니처가 표식이에요.
오답! Express는 함수 이름이나 내용을 보지 않아요 — 오직 매개변수 개수로 구분합니다. 4개(첫 번째가 err)면 에러 미들웨어, 3개면 일반 미들웨어. 이게 유일한 표식이라서 일반 미들웨어를 만들 때는 절대 4개로 쓰면 안 됩니다.

5회원이 자기 프로필에서 닉네임만 바꾸는 API를 만들려고 합니다. 이메일이나 비밀번호는 건드리지 않습니다. HTTP 메서드는 PUT과 PATCH 중 어느 쪽이 더 적합하고, 그 이유는?

PUT — 어떤 수정이든 PUT을 쓰는 게 REST의 원칙이다
PATCH — 리소스의 일부 필드만 수정하는 의미라 적합하다 (PUT은 전체 교체의 의미라 닉네임 외의 필드도 함께 보내야 하는 부담이 있음)
PUT — PATCH는 GET처럼 데이터 조회에만 쓰는 메서드다
상관없다 — PUT과 PATCH는 의미가 완전히 동일하다
정답! PUT은 "전체 교체"의 의미라 모든 필드를 보내야 한다는 뉘앙스가 있고, PATCH는 "일부 수정"의 의미예요. 닉네임만 바꿀 때는 PATCH가 의도를 더 정확히 표현해줍니다. 실제로 우리 프로젝트의 글 수정 API도 일부 필드만 받는 PATCH로 구현돼 있어요(posts.ts의 router.patch('/:id', ...)).
오답! PUT과 PATCH는 의미가 달라요. PUT = 전체 교체(모든 필드를 다시 받아 통째로 갈아끼움), PATCH = 일부 수정(받은 필드만 변경). 닉네임 하나만 바꿀 땐 PATCH가 의도에 정확히 맞습니다. PATCH는 조회용이 아니에요(조회는 GET).
