API 연동 기초 — fetch · async/await · CORS · onMounted

1다음 코드에서 console.log(result)는 무엇을 출력할까요?

const result = fetch('http://localhost:3000/api/boards')
console.log(result)
서버에서 받은 게시판 배열 (예: [{id:1,...}])
undefined
Promise { <pending> } — 약속 쪽지 (아직 결과 없음)
에러: TypeError
정답! fetch는 호출 즉시 Promise를 반환합니다. await 없이 받으면 아직 응답이 도착하지 않은 "pending" 상태의 Promise만 손에 쥐게 돼요. JavaScript는 응답을 기다리지 않고 다음 줄로 넘어갑니다.
오답! fetch는 응답이 올 때까지 기다리지 않아요. 즉시 Promise(약속 쪽지)를 반환하고, await로 명시적으로 기다리지 않으면 pending 상태의 Promise가 그대로 변수에 담깁니다.

2JavaScript가 fetch 같은 네트워크 요청을 비동기로 처리하는 가장 큰 이유는?

동기 방식이 기술적으로 불가능해서
브라우저의 JS는 싱글 스레드라, 동기로 기다리면 그 동안 화면 전체가 멈추기 때문
비동기가 동기보다 항상 빠르기 때문
서버가 동기 요청을 받지 못하기 때문
정답! 브라우저 JS는 일꾼이 한 명(싱글 스레드)이라 그 일꾼이 응답을 기다리며 멈추면 클릭, 스크롤, 애니메이션 등 모든 UI가 함께 얼어붙어요. 그래서 오래 걸리는 작업은 비동기로 설계된 거예요.
오답! 핵심은 "싱글 스레드"예요. JS 일꾼이 응답을 기다리며 멈추면 화면이 통째로 얼어붙기 때문에, 네트워크처럼 오래 걸리는 일은 비동기로 처리하도록 설계됐어요.

3http://localhost:5173 에서 실행되는 Vue 앱이 http://localhost:3000/api/boards 로 fetch했을 때 "Failed to fetch / CORS" 에러가 나는 이유는?

서버가 다운되어 있어서
URL의 철자가 틀려서
백엔드가 HTTPS가 아니어서
포트가 다르면 출처(origin)가 달라서, 브라우저가 JS의 cross-origin 요청을 기본적으로 차단하기 때문
정답! 출처(origin)는 프로토콜+호스트+포트로 구성되고, 셋 중 하나라도 다르면 다른 출처입니다. 5173과 3000은 포트가 다르므로 다른 출처고, 브라우저는 보안상 JS의 cross-origin 요청을 기본 차단합니다. 서버가 CORS 헤더로 명시적으로 허락해줘야 통과돼요.
오답! CORS 에러의 원인은 "출처가 다름"이에요. 프로토콜·호스트·포트 중 하나라도 다르면 다른 출처고, 브라우저는 보안상 JS의 cross-origin 요청을 기본적으로 막습니다. 서버가 cors 미들웨어로 허락해줘야 풀려요.

4다음 코드에서 boards.value = data 줄을 지우면 어떻게 되나요?

const boards = ref<any[]>([])

onMounted(async () => {
  const response = await fetch('http://localhost:3000/api/boards')
  const data = await response.json()
  boards.value = data   // ← 이 줄을 지우면?
})
에러가 나면서 페이지가 멈춘다
fetch는 정상적으로 일어나지만, boards가 빈 배열 그대로라 화면에 아무것도 안 나온다
data가 자동으로 boards에 들어간다 (Vue가 알아서 처리)
서버에서 데이터를 받아오지 못한다
정답! 데이터를 "가져오는 것"과 "화면에 반영되는 것"은 별개예요. ref.value에 새 값을 대입해야 Vue의 반응성 시스템이 작동해서 화면을 다시 그립니다. 지역 변수 data에만 담아두면 아무 일도 안 일어나요.
오답! fetch는 잘 일어나고 data 변수에도 값이 들어가지만, ref인 boards는 그대로 빈 배열이에요. Vue는 ref의 .value가 바뀔 때만 화면을 다시 그리기 때문에, 대입하지 않으면 화면이 업데이트되지 않습니다.

5fetch에 대한 설명 중 맞는 것은?

서버가 404나 500을 반환하면 fetch의 Promise가 자동으로 reject되어 catch로 빠진다
fetch는 응답 본문을 자동으로 JSON으로 파싱해서 돌려준다
fetch는 4xx/5xx를 에러로 보지 않으므로, response.ok로 직접 확인해서 throw해야 catch로 흐른다
fetch는 await가 필요 없다 (동기로 동작한다)
정답! 이건 진짜 자주 실수하는 함정이에요. fetch 입장에선 "응답을 받았다"는 사실 자체가 성공이라, 4xx/5xx도 정상 응답으로 취급해요. 그래서 if (!response.ok) throw new Error(...) 패턴으로 직접 챙겨야 catch 블록으로 흐릅니다.
오답! fetch는 4xx/5xx를 에러로 취급하지 않아요. 응답을 받았다는 것 자체를 성공으로 봅니다. 그래서 response.ok를 직접 확인해서 throw해줘야 catch 블록으로 흘려보낼 수 있어요. .json()도 별도로 await 해야 본문이 파싱돼요.

6아래 패턴에서 finally 블록의 역할은?

try {
  const res = await fetch('/api/boards')
  boards.value = await res.json()
} catch (err) {
  error.value = '에러'
} finally {
  loading.value = false
}
에러가 났을 때만 실행된다
성공했을 때만 실행된다
try 블록보다 먼저 실행된다
성공이든 실패든 무조건 실행되어, 로딩 상태를 항상 꺼지게 보장한다
정답! finally는 try가 성공했든 catch로 빠졌든 무조건 실행돼요. 그래서 "성공/실패와 무관하게 반드시 해야 할 정리 작업"에 딱 맞아요. 로딩 상태 끄기, 연결 닫기, 임시 파일 지우기 같은 데 자주 쓰입니다.
오답! finally는 try가 성공했든 catch로 빠졌든 무조건 실행돼요. 그래서 "어떤 경우에도 반드시 해야 할 정리"에 적합해요. 로딩 상태 끄기가 대표 예시예요.
