Chapter 24 Quiz

배포하기 — SQLite→PG 마이그레이션 · Vercel · Render

1같은 SQL을 실행하는데 better-sqlite3와 pg(PostgreSQL)가 코드 모양이 다른 이유는?

// before (SQLite)
const user = db.prepare('SELECT * FROM users WHERE id = ?').get(1)

// after (pg)
const result = await pool.query('SELECT * FROM users WHERE id = $1', [1])
const user = result.rows[0]
PostgreSQL은 SQL을 못 읽어서 JS 코드로 변환해야 한다
pg는 더 새로 나온 라이브러리라 작성자가 멋있게 짠 것
SQLite는 같은 컴퓨터 안의 파일이라 즉시(동기) 응답 가능하지만, PostgreSQL은 네트워크 너머의 서버라 응답을 기다려야(비동기/await) 함 — 그래서 결과 형태도 다름
TypeScript를 써서 타입이 다르게 보이는 것뿐, 실제 동작은 동일
정답! 핵심은 "데이터베이스가 어디 있는가"예요. SQLite는 같은 컴퓨터의 파일을 직접 읽고 쓰니 함수가 바로 결과를 줄 수 있어요(동기). PostgreSQL은 네트워크 건너편의 별도 서버라 요청→응답 왕복이 필요하고, 그동안 다른 일을 못하면 안 되니 await로 기다리는 비동기 패턴을 씁니다. 결과도 row뿐 아니라 메타데이터를 함께 담은 객체(`result.rows`)로 와요.
오답! 진짜 이유는 "DB가 같은 컴퓨터에 있냐, 네트워크 너머에 있냐"입니다. SQLite는 로컬 파일 → 동기(즉시 응답). PostgreSQL은 원격 서버 → 비동기(await로 응답 대기). 코드 스타일이 멋이거나 임의 규칙이 아니라, 통신 모델이 달라서 강제되는 것이에요.

2PostgreSQL 쿼리에서 ? 대신 $1, $2를 쓰는 이유로 가장 정확한 것은?

// SQLite
db.prepare('INSERT INTO posts (title, content) VALUES (?, ?)').run(t, c)

// PostgreSQL
await pool.query('INSERT INTO posts (title, content) VALUES ($1, $2)', [t, c])
PostgreSQL은 보안 기능이 없어서 번호를 직접 매겨야 한다
DB마다 자리표시자(placeholder) 방언이 다르기 때문 — SQLite/MySQL은 ?, PostgreSQL은 $1, $2로 위치를 명시. 둘 다 SQL 인젝션 방지라는 같은 목적
?는 PostgreSQL에서 예약어라 컬럼명으로 못 써서
JavaScript의 템플릿 리터럴(${"${...}"}) 표기에서 따온 것
정답! 자리표시자는 사용자 입력을 SQL과 안전하게 분리하는 보안 장치예요(SQL 인젝션 방지). 목적은 같지만 표기법은 DB마다 달라요 — SQLite/MySQL은 위치 기반 ?, PostgreSQL은 번호 기반 $1, $2(같은 값 재사용 가능), Oracle은 이름 기반 :name. 23챕터 퀴즈에서 본 SQL 방언의 또 다른 예입니다.
오답! 자리표시자의 본질적 목적은 같습니다 — 사용자 입력을 안전하게 SQL에 끼워넣어 인젝션을 막는 것. 단지 표기법이 DB마다 달라요(SQL 방언). PG는 번호로 위치를 명시(같은 값 여러 번 쓰기 편함). 보안 부재나 예약어 문제는 아니에요.

3왜 프론트엔드는 Vercel에 올리고, 백엔드는 Render에 올렸을까요?

두 서비스 다 무료라 비용을 분산하기 위해
프론트는 빌드 결과(HTML/JS/CSS)만 있으면 되는 정적 호스팅(서버 실행 불필요)이라 Vercel에 적합. 백엔드는 Express 서버가 24시간 떠서 요청을 받아야 해서 서버 실행 환경(Render)이 필요
Vercel은 Vue 전용, Render는 Node.js 전용 서비스라서 강제됨
한 서비스에 다 올리면 보안상 위험해서 분리해야 한다
정답! 본질은 "실행 모델의 차이"예요. 프론트는 사용자 브라우저가 다운로드해서 직접 실행하는 정적 파일 묶음 → 그저 잘 분배해주는 CDN(정적 호스팅)이면 충분. 백엔드는 우리가 작성한 Express 코드가 클라우드 컴퓨터 위에서 항상 켜져 있어야 요청을 받아 처리할 수 있음 → 진짜 컴퓨터(서버 실행 환경)가 필요. Vercel/Render는 서로 다른 모델에 특화된 서비스를 무료로 제공할 뿐.
오답! 핵심은 비용/브랜드/보안이 아니라 "실행 모델"이에요. 프론트는 빌드 결과(JS/HTML/CSS)를 사용자 브라우저로 그냥 보내주면 끝(정적 호스팅, 서버 안 켜져 있어도 OK). 백엔드는 우리 코드가 24시간 클라우드에서 켜져 있어야 동작(서버 실행 환경 필요). 두 요구가 다르니 적합한 서비스를 선택한 것.

4Render 무료 티어에서 첫 요청이 30초~1분 정도 느린 현상(콜드 스타트)이 나타나는 이유는?

PostgreSQL 비밀번호가 매번 자동 변경돼서 인증이 다시 일어나기 때문
사용자 브라우저가 캐시를 매번 비우기 때문
무료 티어는 15분 무요청이면 서비스를 잠재워(spin down) 자원을 회수함. 다음 요청이 오면 컨테이너를 다시 켜고 npm start로 서버를 띄우는 시간이 걸려서
Vercel과 Render 사이의 네트워크 거리 때문
정답! 무료 티어는 자원이 한정돼서 "안 쓰면 끔" 정책을 씁니다. 15분 동안 요청이 없으면 컨테이너를 종료(spin down)해 다른 사용자에게 자원을 양보. 다음 요청이 오면 다시 컨테이너를 띄우고 Node 프로세스 + DB 연결까지 초기화하느라 시간이 걸림. 유료 플랜으로 업그레이드하면 항상 켜진 상태가 유지돼요. 운영 서비스에서는 ping 스케줄러 등으로 우회하기도 해요.
오답! 콜드 스타트의 원인은 "무료 티어의 비활성 시 종료" 정책이에요. 비밀번호 변경, 브라우저 캐시, 네트워크 거리 같은 건 매 요청마다 영향을 주는 요소가 아니에요. Render 무료 티어는 15분 무요청 → 컨테이너 종료, 다음 요청 시 다시 시동(빌드는 안 다시 함, 그저 실행만 다시) → 그 시간이 30초~1분.

5Render에 JWT_SECRET 환경변수를 등록하지 않았더니 로그인 시 500 에러("secretOrPrivateKey must have a value")가 발생했어요. 이 사건이 알려주는 것은?

const token = jwt.sign(
  { userId: user.id, email: user.email },
  process.env.JWT_SECRET!,   // ← undefined면 에러
  { expiresIn: "24h" },
)
로컬에서만 환경변수를 쓰면 되고, 배포 환경에는 따로 등록할 필요 없음
.env는 Git에 안 올라가므로(.gitignore), 배포 환경에는 우리가 따로 등록해야 함. 로컬 OK ≠ 배포 OK — 환경별 점검이 필수
JWT 라이브러리에 버그가 있어서 발생한 일시적 현상
PostgreSQL이 JWT_SECRET을 자동으로 관리해주는 기능을 켜야 함
정답! 환경변수 분리(23챕터)의 진짜 이유가 24챕터에서 드러난 거예요. .env는 비밀이라 Git에 안 올라가니 → 배포 서버는 그 값을 모름 → 우리가 직접 등록해야 함. 등록을 빼먹으면 process.env.JWT_SECRET가 undefined가 되고 jwt.sign이 즉시 에러. "로컬에서 됐으니 배포에서도 되겠지"는 환경변수 누락으로 자주 깨집니다. 배포 후에는 환경변수 체크리스트를 반드시 점검해야 해요.
오답! 이건 코드 버그도, DB 기능도, 일시적 현상도 아니에요. 환경변수의 본질을 보여주는 사건입니다. .env는 .gitignore로 제외되니 배포 서버는 값을 모름 → 우리가 클라우드 콘솔(Render Environment 탭)에서 직접 등록해야 함. 누락하면 process.env.X가 undefined가 돼서 의존하는 코드가 깨져요. "로컬 동작 ≠ 배포 동작"이라는 큰 교훈.
